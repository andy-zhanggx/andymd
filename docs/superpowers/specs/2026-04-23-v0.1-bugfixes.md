## Issue A: 单文件打开路径不可发现 + `.md` 在 Open 对话框灰显

### Symptom

1. Sidebar 只暴露 "Open…" 按钮用于打开**工作区**；想打开**单个 `.md` 文件**时用户没有视觉入口（功能在 `⌘O` 和 macOS 菜单 File → Open File… 里，但不可见）。
2. 当用户真的走 `⌘O` 或菜单触发原生对话框时，`.md` 文件在 NSOpenPanel 里**灰显不可选**。

### Root cause

1. `WorkspaceSwitcher` 只调用 `pickWorkspaceDir`；没有 "open single file" 的 UI entry。空状态占位文字也只是提示按 `⌘O`，不是可点击按钮。
2. `pick_markdown_file` 的 filter 只声明了 `["md", "markdown"]`。macOS NSOpenPanel 现代 API 通过 UTI 而非扩展名过滤；`.md` 没有系统默认 UTI 注册（除非 app 的 Info.plist 声明为 document type），NSOpenPanel 因此把它们归为"非 Markdown 类"灰掉。

### Fix

1. `src/components/Sidebar/WorkspaceSwitcher.tsx`：在 "Open…" 按钮旁边新增 "Open File" 按钮，调用 `dialogService.pickMarkdownFile()` → `useDocumentStore.getState().open(path)`。
2. `src/components/Editor/MarkdownEditor.tsx`：空状态占位改为两个居中可点击按钮（Open File / Open Workspace）。
3. `src-tauri/src/commands/workspace_cmd.rs` 的 `pick_markdown_file`：把 filter 扩展为 `["md", "markdown", "mdown", "mkd", "txt"]`，并追加一个 `"All Files", &["*"]` 作为兜底，用户始终能选到任意文件。
   - **Note**：此文件当前已有 Claude 在纪律更新前的直接编辑（workspace_cmd.rs 脏态中）。本 spec 正式接管该改动，所以会一并走 Codex dispatch 回滚 + 重做流程（或直接 stage 当前改动并加上按钮补充——见"Implementation"）。

### Verification

- 重启 `pnpm tauri dev`
- 无工作区状态：编辑器中央显示两个按钮；点 "Open File" → 原生对话框 → `.md` 不灰显 → 选中 → 内容载入
- 有工作区状态：Sidebar 顶部 "Open File" 按钮能独立触发单文件对话框，且不影响当前工作区
- 选到 `.txt` 或其他文本也能打开（All Files 兜底）

## Issue B: Milkdown 显示 Markdown 原文（B2），WYSIWYG 未生效

### Symptom

打开 `.md` 文件后，编辑器区显示的是 Markdown 源码（`#`、`**`、`` ` `` 等标记符号原样可见），而不是 WYSIWYG 渲染（h1 字体加大、bold 变粗等）。

### Root cause（待 Codex 实际诊断收敛）

主要怀疑三条，按优先级：

1. **React 18 StrictMode 竞态 + `editor.create()` 未 await**
   `editor.create()` 返回 Promise。当前代码 fire-and-forget。StrictMode 下 effect 跑两遍：
   - 第一次 setup：`create()` 异步挂载 ProseMirror view 到 `root`
   - 第一次 cleanup：`disposed=true`, `root.innerHTML=''`（这会清空第一次正在挂载的 DOM）
   - 第二次 setup：新建一个 editor，但第一次 Promise 仍在后台进行，两个实例可能对同一个 root 起争
   - 结果：view 未正确挂载或 selection/schema 状态错乱，文本以 contenteditable 纯字符形式残留
2. **Milkdown 7.20 API 变更**
   `defaultValueCtx` 在 7.x 里仍支持 string，但 `editorViewOptionsCtx` 可能需要不同 shape，且 `nord` theme 的 CSS import 路径在某些版本是 `@milkdown/theme-nord/style.css` 而在另一些是 `@milkdown/theme-nord/dist/style.css`。如果 CSS 没成功加载，ProseMirror 会渲染成无样式 DOM，视觉上接近"原文"。
3. **ProseMirror view CSS 缺失**
   CommonMark/GFM preset 只定义 schema 和序列化器，不自带 `.ProseMirror` 的基础 CSS。Milkdown 依赖 theme 包提供。如 theme 没生效，`<h1>` 会变成无 `font-size` 的普通 div 看起来像字符流。

### Fix 方向

按"先最小改动"顺序 Codex 尝试：

1. **Step 1 — await create + 保护 StrictMode**：
   ```tsx
   let editor: Editor | undefined;
   (async () => {
     const e = await buildEditor({...}).create();
     if (disposed) { e?.destroy?.(); return; }
     editor = e;
     rewrite();
   })();
   return () => {
     disposed = true;
     mo.disconnect();
     scroller?.removeEventListener('scroll', scrollHandler);
     // flush session
     editor?.destroy?.().catch(() => {});
     root.innerHTML = '';
   };
   ```
   **预期**：修复竞态。如果 B2 是 H1 造成的，这一步就解决。
2. **Step 2 — 显式加载 ProseMirror 基础 CSS**：
   如果 Step 1 后仍 B2，在 `editor-styles.css` 里追加最小 ProseMirror 样式（或确保 `@milkdown/theme-nord/style.css` 确实被 Vite 打包了——检查 dev build）：
   ```css
   .milkdown { outline: none; }
   .milkdown .ProseMirror { min-height: 100%; }
   .milkdown h1 { font-size: 2em; font-weight: 600; margin: 0.67em 0; }
   .milkdown h2 { font-size: 1.5em; font-weight: 600; margin: 0.83em 0; }
   .milkdown h3 { font-size: 1.17em; font-weight: 600; margin: 1em 0; }
   .milkdown strong { font-weight: 700; }
   .milkdown em { font-style: italic; }
   .milkdown code { background: var(--code-bg); padding: 0 4px; border-radius: 3px; font-family: 'SF Mono', Menlo, monospace; font-size: 0.9em; }
   .milkdown pre { background: var(--code-bg); padding: 12px; border-radius: 6px; overflow: auto; }
   .milkdown blockquote { border-left: 3px solid var(--border); padding-left: 12px; color: var(--fg-secondary); margin: 0 0 1em 0; }
   .milkdown ul, .milkdown ol { padding-left: 1.5em; }
   .milkdown table { border-collapse: collapse; margin: 1em 0; }
   .milkdown th, .milkdown td { border: 1px solid var(--border); padding: 4px 8px; }
   ```
3. **Step 3（fallback）** — 若前两步都不足，切到 `@milkdown/crepe`（一个"装好电池"的 Milkdown 预设）或直接降级到 CodeMirror 6 装饰器方案。这属于架构级改动，超出本 bug fix 范围，另开 spec。
